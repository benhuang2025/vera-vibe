# Brevis Vera Spec v1: Data Structures & Core Interfaces

## 1. Core Models

All data structures should support Serde serialization for cross-module transmission.

### 1.1 PhotoMetadata
Records the original metadata of the photograph.
```rust
struct PhotoMetadata {
    device_id: String,      // Simulated unique device identifier
    timestamp: u64,         // Capture timestamp
    width: u32,             // Original width
    height: u32,            // Original height
    image_hash: [u8; 32],   // SHA-256 hash of the original image
}
```

### 1.2 Signature
Stores ECDSA P-256 signature information.
```rust
struct Signature {
    r: [u8; 32],            // Signature component r
    s: [u8; 32],            // Signature component s
    public_key: Vec<u8>,    // Corresponding public key (SEC1 format)
}
```

### 1.3 SignedPhoto
The complete original signed package.
```rust
struct SignedPhoto {
    image_bytes: Vec<u8>,
    metadata: PhotoMetadata,
    signature: Signature,
}
```

### 1.4 EditOperation
Enum defining supported edit transformations.
```rust
enum EditOperation {
    Crop { x: u32, y: u32, width: u32, height: u32 },
    AdjustBrightness { delta: i16 }, // Range: -255 to 255
}
```

### 1.5 EditManifest
An ordered set of editing instructions.
```rust
struct EditManifest {
    operations: Vec<EditOperation>,
}
```

### 1.6 ProofPackage
The final artifact sent to the verifier.
```rust
struct ProofPackage {
    edited_image: Vec<u8>,
    proof: Vec<u8>,          // Proof generated by Pico ZKVM
    public_values: PublicValues,
}

struct PublicValues {
    pub_key_hash: [u8; 32],  // Hash of the public key used in the proof
    edit_types: Vec<String>, // List of operation types (parameters hidden)
    output_image_hash: [u8; 32], // Hash of the final edited image
}
```

## 2. Module Interfaces

### 2.1 Mock Signer (Capture Layer)
Simulates camera capture and signing.
- `fn generate_keypair() -> (SecretKey, PublicKey)`
- `fn sign(image: &[u8], device_id: &str) -> SignedPhoto`
- `fn verify_signed_photo(signed: &SignedPhoto) -> bool`

### 2.2 Editor (Editing Layer)
Handles UI-side or local editing.
- `fn apply_manifest(image: &[u8], manifest: &EditManifest) -> Vec<u8>`
- `fn crop(image: &DynamicImage, x, y, w, h) -> DynamicImage`
- `fn adjust_brightness(image: &DynamicImage, delta: i16) -> DynamicImage`

### 2.3 ZK Guest Program (app/)
Code running inside Pico ZKVM.
- **Inputs (Private)**: `SignedPhoto`, `EditManifest`
- **Logic**:
    1. Re-hash `SignedPhoto.image_bytes` and compare with `metadata.image_hash`.
    2. Verify the ECDSA P-256 signature using `metadata.image_hash` and `signature`.
    3. Sequentially execute operations in `EditManifest`.
    4. Compute the hash of the final result pixels.
- **Commit (Public)**: `PublicValues` (Public key hash, edit types, result hash).

### 2.4 Prover
Host-side program that initializes and runs the ZKVM.
- `fn generate_proof(signed: SignedPhoto, manifest: EditManifest) -> ProofPackage`

### 2.5 Verifier
Independent verification endpoint.
- `fn verify(pkg: ProofPackage, trusted_keys: &[PublicKeyHash]) -> bool`

---

## 3. Storage & Transmission Formats

- **Mock Data**: Stored as `.json` files.
- **Intermediate Images**: Use `DSC00050.JPG` as the default source; output as `.png` to avoid hash mismatches caused by lossy compression.
- **Proofs**: Serialized as binary `.bin` or within JSON.
