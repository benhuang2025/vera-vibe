# Brevis Vera — 核心架构 Q&A

## Q1: C2PA 的验证是在 ZKVM Program 里吗？如果不是，我们怎么确保输入到 Pico ZKVM Program 里的图片就是外部 C2PA 验证过的图片？

**答案：C2PA（或 mock ECDSA P-256）签名验证必须在 ZKVM 内部执行。**

### 原因

如果签名验证发生在 ZKVM 外部（比如在普通 Rust 代码里），那么 ZK Proof 只能证明："我收到了一堆像素，并对它们进行了裁剪"。它**无法证明**这堆像素真的来自那张经过硬件签名的原始照片。任何人都可以伪造一张图片，声称"我在外面验过签名了"，然后把假图片丢进 ZKVM。

### 实现方式

我们将**带有签名和元数据的原始图片**作为 ZKVM 的私密输入（Private Witness）。ZKVM 的 Guest Program 逻辑包含：

1. **解析签名元数据**：提取签名值 (r, s) 和原始图片的哈希。
2. **签名验证**：在 ZKVM 电路里运行 ECDSA P-256 验证算法，用公钥验证签名是否合法。
3. **哈希一致性**：验证签名所保护的哈希值与传入的原始图片字节的 SHA-256 是否完全匹配。

只有这三步在 ZKVM 内部全部通过，生成的 Proof 才有密码学意义——它证明了："这段逻辑是基于一张**经硬件签名验证通过**的真实图片运行的"。

签名者的公钥（或其哈希）会作为 **Public Output** 被 commit 出来，这样验证者可以检查该公钥是否属于受信任的设备（如 Sony 相机）。

## Q2: 如果我们提供一个界面做裁剪等操作，然后保存了一张图片，后面要 Pico 来证明界面上的操作，这个如何证明？是否需要在 ZKVM 里重走一遍？

**答案：是的。这在 ZK 证明中叫"确定性重放"（Deterministic Replay）。**

### 原因

直接对比"编辑前"和"编辑后"两张图片是很难用 ZK 证明其"合规性"的——因为像素差异太模糊，ZKVM 无法从差异中反推出"这些变化是合法裁剪造成的，而不是 AI 换脸造成的"。

所以标准做法是：**不比较图片本身，而是重放操作过程。**

### 完整流程

```
┌─────────────────────────────────────────────────────────────┐
│  UI 层（用户可见）                                            │
│                                                             │
│  1. 用户上传原始签名图片                                       │
│  2. 用户在界面上操作（拖动裁剪框、调整亮度等）                    │
│  3. UI 记录操作参数：[Crop(x:100, y:100, w:500, h:500)]       │
│  4. UI 调用 Photon 等库实时生成预览，保存编辑后图片               │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼ 提交：原始图片 + 签名 + 操作参数列表
┌─────────────────────────────────────────────────────────────┐
│  ZKVM 层（Pico Guest Program）                               │
│                                                             │
│  1. 验证原始图片的 ECDSA P-256 签名 → 确保来源真实              │
│  2. 按照操作参数列表，在内存中对原始图片重新执行相同操作            │
│  3. 计算重放结果图片的 SHA-256 哈希                             │
│  4. commit Public Output:                                    │
│     - 签名者公钥哈希（证明来源）                                │
│     - 编辑操作类型列表（证明做了什么类型的编辑）                  │
│     - 编辑后图片的哈希（证明结果一致）                           │
└──────────────────────┬──────────────────────────────────────┘
                       │
                       ▼ 输出：Proof + Public Output
┌─────────────────────────────────────────────────────────────┐
│  验证层（Verifier）                                          │
│                                                             │
│  1. 调用 Pico SDK 验证 Proof 的数学正确性                      │
│  2. 独立计算收到的编辑后图片的 SHA-256 哈希                     │
│  3. 对比：自己算的哈希 == Proof Public Output 中的哈希？         │
│  4. 检查公钥是否属于受信任的设备                                │
│  5. 输出最终判定：✅ Verified 或 ❌ Rejected                   │
└─────────────────────────────────────────────────────────────┘
```

### 关键要点

- **哈希比对发生在验证者（Verifier）侧**，不在 ZKVM 内部。ZKVM 只负责 commit 哈希值作为 Public Output。验证者收到编辑后图片后自己算一遍 SHA-256，和 Proof 的 Public Output 比对。

- **编辑操作的具体参数（如裁剪坐标）是私密的**。Public Output 中只暴露操作"类型"（如"进行了裁剪和亮度调整"），不暴露具体参数（如"裁剪了哪个区域"）。这符合 Product Brief 中 "hide the details of such an operation" 的要求。

- **编辑操作必须是确定性的**。同样的输入 + 同样的参数 = 必须产出完全相同的像素。这意味着不能使用带随机性的滤镜（如随机噪点），或者需要将随机种子也作为参数传入。

- **这就是为什么编辑操作要足够简单**。因为在 ZKVM 里每重走一遍像素处理，消耗的算力（Constraints数量）都比普通 CPU 高出成千上万倍。

### 这个设计的密码学闭环

| 属性 | 由谁保证 |
|------|---------|
| 图片来自真实设备 | ZKVM 内签名验证 |
| 只进行了声称的编辑 | ZKVM 内确定性重放 |
| 没有额外篡改 | 哈希比对（重放结果 vs 实际图片）|
| 隐藏编辑细节 | ZK 的零知识性（参数是 Private Witness）|
| 证明不可伪造 | ZK Proof 的 Soundness 性质 |
