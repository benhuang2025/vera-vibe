# 🔍 Multi-CPU Parallel Proving: Initial Profiling Analysis

在本阶段的基准测试中，我们观察到 64 核并行处理 **30 亿 Cycles** 的耗时约为 **1 分钟**。虽然相比单核的 **280 秒** 有显著提升，但并未达到理论上的线性加速（即缩短至数秒）。以下是性能损耗的核心原因分析：

## 1. 冗余的 ECDSA 验证 (Redundant Cryptography)
*   **现象**: 当前版本中，64 个分片共享同一个 `app.elf`（包含完整的签名验证逻辑）。
*   **开销**: 每一个核心都在独立运行一次 P-256 签名验证（约 2400 万 Cycles/次）。
*   **损耗**: 总计浪费了约 **15 亿 Cycles** 的算力在执行完全相同的验证逻辑上，这极大地拉平了并行的收益曲线。

## 2. IO 与序列化瓶颈 (Data Serialization Overhead)
*   **现象**: 原图 JSON 数据量高达 **312MB**。
*   **开销**: `prover` 在启动每个分片时，需要将这 312MB 的 JSON 反序列化 64 次。
*   **瓶颈**: JSON 解析是非常消耗 CPU 的串行或半并行任务。内存总线在频繁搬运这 300MBx64 的数据时会产生严重的竞争（Lock Contention）。

## 3. ZKVM 启动与初始化开销 (Startup Latency)
*   **现象**: 每个线程都在独立初始化 `DefaultProverClient` 并加载 330KB 的 ELF 文件。
*   **损耗**: 虽然单个 ELF 加载很快，但在高并发场景下，文件系统 I/O 和环境准备的累积延迟变得不可忽视。

## 4. 内存带宽墙 (The Memory Wall)
*   **现象**: 124 个 CPU 核心共享有限的内存通道。
*   **瓶颈**: ZKVM 的模拟执行伴随着频繁的内存读写。当 64 路并发同时全速访问内存时，带宽会达到饱和，导致每个核心的指令吞吐量大幅下降。

---

## 🚀 性能跳跃：Shard-Aggregator 架构改进
为了解决上述瓶颈，下一阶段我们将实施 **“轻量化分片证明 (Shard App)”** 方案：

1.  **极简化 Guest (Shard App)**: 剔除 JSON 解析、剔除 ECDSA 验证，仅保留核心哈希逻辑。
2.  **二进制直传 (Raw Bytes)**: 通过 `stdin` 直接传递原始像素切片（约 500KB），规避 JSON 系统开销。
3.  **单次验证 (Aggregator)**: 所有的签名校验逻辑移至最后的聚合阶段，仅运行 1 次。

**预期目标**: 将延迟从分钟级优化至 **10-15 秒** 级别。
